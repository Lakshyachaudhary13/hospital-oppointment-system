### Slide 3: The Problem Landscape
- **Manual Overhead:** High error rates in scheduling and slot overlapping.
- **Discovery Bottlenecks:** Patients' inability to filter doctors by verified experience or specialty.
- **Data Fragmentation:** Consultation histories trapped in physical physical silos.
- **Operational Lag:** Manual registration causing massive reception-area delays.

> **Speaker Note:** "The current landscape is plagued by fragmentation. Patients often struggle to find the right specialist, while hospital administrators face the logistical nightmare of manual scheduling conflicts."

### Slide 5: Technical Architecture (SRS)
- **Frontend Layer:** HTML5, CSS3 (Glassmorphism), JavaScript (Asynchronous APIs).
- **Control Layer:** Java 17 | Spring Boot Framework (RESTful Architecture).
- **Data Layer:** SQL-based Relational Database (H2/PostgreSQL).
- **Engineering Tools:** VS Code, Git, Maven Built-automation.

> **Speaker Note:** "Technically, we've utilized a 3-tier enterprise architecture. We leverage Spring Boot for robust security and business logic, while the frontend uses modern glassmorphism design to ensure a high-end user experience."

---

### Slide 6: Master System Modules
- **Authentication Core:** Secure JWT-based logic for private data protection.
- **Specialist Directory:** A dynamic, searchable database of medical experts.
- **Booking Engine:** The logical core that manages slot selection and verification.
- **Actionable Analytics:** An Admin Dashboard visualizing hospital patient metrics.

> **Speaker Note:** "The system is modular. From the secure login experience to the searchable directory and the live stats dashboard, each module is designed to work in harmony under a single interface."

---

### Slide 7: Database Modeling
- **Entities:**
    - **User Schema:** Centralizing Identity and Role-based permissions.
    - **Doctor Schema:** Persisting expertise, experience, and professional imaging.
    - **Appointment Schema:** Mapping time-slots to patient-doctor relationships.
- **Integrity:** Enforcing 1-to-many relationships to ensure data consistency.

> **Speaker Note:** "Our data model ensures integrity. By mapping doctors and patients to specific time-slots with unique IDs, we've created a reliable audit trail for every consultation in the hospital."

---

### Slide 8: Enterprise Topology
- **Workflow:** UI Requests → Controller Logic → JPA Persistence → SQL Data.
- **Interchange:** Standardized JSON formatting for all API communications.
- **Scalability:** Built to handle hundreds of concurrent users through optimized Spring Boot threading.

> **Speaker Note:** "The system topology follows industry best practices. By decoupling the presentation layer from the data logic, we ensure that CarePlus is both scalable and easy to maintain as feature requirements grow."
